# 叩问录 — AI 对话知识整理指南

## 这是什么

这是一份给 Claude（或其他 AI 模型）的工作指南。当用户提供一段 AI 对话的原始聊天记录并说「添加」时，你需要按照本文档的规则，将这段对话提炼为一个结构化的知识条目，并输出为符合产品数据格式的 JSON 对象。

---

## 产品理念

**叩问录**是一个将 AI 对话转化为可回顾的知识文档的阅读产品。

核心洞察：用户与 AI 的每一次对话，本质上是一次**通过提问获得认知跃迁**的过程。传统笔记（Notion、飞书文档）之所以吃灰，是因为打开它的成本太高——你不记得里面有什么，标题唤不起当时的思考状态。

**叩问录**的解决方案：以**用户的问题链**作为文章的骨架和入口。用户第一眼看到的不是一篇文章，而是自己当时问了什么。看到问题就能瞬间回忆起当时的困惑和好奇心，从而产生重新阅读的欲望。

---

## 数据结构

每篇文章独立存储在 `src/articles/` 目录下（如 `01.ts`、`02.ts`），由 `src/articles/index.ts` 汇总导出 `ARTICLES` 数组（UI 组件在 `src/App.tsx`，两者已解耦）。每个文章文件导出一个符合 `Article` 类型的对象：

```ts
{
  id: number,                // 递增编号，取当前最大值 +1
  meta: {
    source: string,          // 来源，如 "Claude"、"Gemini"、"ChatGPT"
    date: string,            // 如 "2025年3月21日"
  },
  title: string,             // 文章标题（15字以内）
  brief: string,             // 一句话概括（50-80字）
  sections: [                // 问答链，1-7个
    {
      question: string,      // 精炼后的核心提问
      context: string | null,// 可选的背景说明（斜体灰色块）
      image?: string,        // 可选的配图路径（相对于 public，如 "/images/xxx.webp"）
      body: string,          // 正文，支持 **加粗**、[文字](链接) 语法，用 \n\n 分段
    },
  ],
  takeaway: string,          // "写在最后"结语
}
```

## 添加文章的步骤

1. 在 `src/articles/` 目录新建文件（如 `20.ts`），`id` 取当前最大值 +1
2. 在 `src/articles/index.ts` 中添加 import 并将其加入 `ARTICLES` 数组
3. 如果有配图，将图片复制到 `public/images/` 目录，在对应 section 中添加 `image: "/images/文件名"`
4. 运行 `npm run build` 验证无报错
5. 首页的篇数和问题数会自动计算，无需手动更新

## 处理流程

当用户提供原始聊天记录并说「添加」（或「处理这段对话」「整理一下」）时，按以下步骤处理：

### 第一步：识别对话结构

1. 区分用户发言和 AI 回复。用户发言通常较短、口语化、包含疑问。
2. 识别对话的来源平台（Claude、Gemini、ChatGPT 等）。
3. 如果聊天记录中有明显的时间戳，提取日期。

### 第二步：提取问题链（最核心的步骤）

**问题链是整个产品的灵魂，必须投入最多精力。**

从用户的所有发言中，提取出推动对话向前发展的**核心提问**。注意：

- **不是每句用户发言都是一个「问题」**。寒暄、确认、补充说明不算。只提取那些引发了新的认知方向的发言。
- **问题要精炼但保留原始锐度**。用户说「这块我不太懂，就是二手的利润怎么会比新的还高」应该精炼为「二手利润竟然高于新机？这不是反直觉吗？」而不是「关于二手利润的问题」。
- **保留追问的递进感**。问题链应该体现用户的认知是如何一步步深入的。
- **通常一篇对话有 2-7 个核心问题**。如果只有 1 轮问答，那就是 1 个问题，不要强行拆分。

### 第三步：检查问题链的完整性（防止遗漏）

这是最容易出错的环节。逐一核对以下清单：

- **对话中是否有「转折点」被遗漏？** 用户从一个话题转向另一个话题、从具体问题上升到方法论、从理论追问到实操——这些转折必须作为独立的问题节点。
- **AI 主动展开的重要分支是否被覆盖？** 有时 AI 的回复中包含了多个重要维度，用户随后追问了其中一个。被追问的那个维度不能丢。
- **用户表达的「反直觉感受」是否被捕获？** 如果用户说「这颠覆了我的认知」「我一直以为不是这样的」「这点我没想到」，这些通常是最有价值的认知节点。
- **对话的最终结论或落脚点是否被包含？** 一段对话的最后一个有实质内容的交换，往往是整个思考的收尾，不能丢。

### 第四步：精炼每个问题对应的回答

将 AI 的原始回复压缩为精炼版本。规则：

- **不是摘要，是「带论据的结论」**。保留关键数据、关键比喻、关键论证链条，砍掉所有寒暄、重复、客套和过渡。
- **保留原文中最有力的表述**。如果 AI 说了一句特别精辟的话（比如「越标准化的商品渠道利润越薄，越非标的商品中间商利润越厚」），原封不动保留。
- **用 `**加粗**` 标记要点**，但不要过度使用。每段正文中加粗不超过 3-4 处。
- **段落之间用空行分隔**（在 JSON 字符串中用 `\n\n`）。
- **单个问题的 body 控制在 150-400 字**。太短没有信息密度，太长则失去精炼的意义。
- **不要使用 markdown 列表格式**（不要用 `- ` 或 `1. `）。用自然的叙述性语言，如需罗列用中文顿号或分号连接。
- **保留专业术语的英文原文**。当原始对话中出现有对应英文的专业术语时，首次出现用「中文（English）」的格式标注，如「点击率（CTR）」「同期群分析（Cohort Analysis）」「自然语言处理（NLP）」。这既保留了术语的精确性，也方便读者后续检索和延伸学习。后续重复出现可只用缩写。

### 第五步：撰写 title、brief 和 takeaway

**title 规则：**
- 15 字以内，有信息量且有吸引力
- 可以用书名号、冒号、破折号
- 示例：「转转商业模式：一门「不可能」的生意是怎么做起来的」

**brief 规则：**
- 50-80 字，一句话说明：从什么出发 → 经历了什么 → 得出了什么
- 示例：「从直觉上的「不可能」出发，逐步拆解了转转的质检壁垒、多层分销和烧钱逻辑。最终发现一个通用规律：越是非标准化、信息不对称的商品，中间商利润越厚。」

**takeaway 规则：**
- 全文最核心的一句话洞察，放在文章末尾的「写在最后」区域
- 应该是那种看完之后会记住的话
- 示例：「核心转换：不要想「怎么推广我的工具」，而是想「我的用户此刻正在哪里喊牙疼」，然后走过去说一句「我刚好是牙医」。」

### 第六步：撰写 context（可选）

context 是问题的前置背景，用于帮助读者理解「为什么会问这个问题」。规则：

- 如果问题本身已经足够清晰，context 设为 `null`
- 如果问题的提出有一个具体的触发点（比如看到了某个数据、对话中前一个回答引出了新的困惑），用 1-2 句话写清楚
- context 在界面上会以灰色斜体显示在问题下方、正文上方

## 质量检查清单

在输出最终结果前，逐项核对：

### 问题链
- 问题数量合理（1-7个，与原始对话的深度匹配）
- 问题之间有递进关系，不是平行罗列
- 没有遗漏关键的认知转折点
- 问题的措辞精炼但保留了原始的锐度和好奇心

### 内容精炼
- 每个 body 都包含关键论据/数据，而不只是结论
- 砍掉了所有寒暄、重复和过渡性语言
- 保留了原文中最精辟的表述
- 单个 body 在 150-400 字之间

### 整体结构
- title 有信息量且简洁（15字以内）
- brief 是一句完整的话（50-80字）
- takeaway 是全文最值得记住的一句话
- 整篇文章从头到尾读起来是一个完整的认知旅程

---

## 特殊情况处理

### 单轮问答
如果对话只有一轮（用户问了一个问题，AI 给了一个回答），那就只有 1 个 section。不要强行拆分成多个问题。

### 超长对话（超过 10 轮）
如果对话非常长，先识别 3-5 个「大的认知阶段」，每个阶段提炼出 1-2 个核心问题。最终控制在 5-7 个问题以内。

### 用户中途换了话题
如果对话前半段和后半段是完全不同的主题，应该拆成两篇独立的文章。

### 对话中有大量代码或技术细节
如果是编程相关的对话，保留关键的技术要点和设计思路，但不需要逐行保留代码。把代码的「为什么这么做」提炼出来，比「具体怎么写」更有回顾价值。

### 对话内容涉及用户的个人情感或心路历程
如果对话中用户表达了深层的纠结、反思或价值观选择（比如「树洞」那篇中创作者在理想与现实之间的挣扎），这些内容必须保留——它们往往是最有回顾价值的部分。

---

## 输出格式

处理完成后，将 JSON 对象作为新条目追加到 `src/articles.ts` 的 `ARTICLES` 数组末尾，然后运行 `npm run build` 验证。

不需要用户提供 id 或 date——id 取当前 ARTICLES 中最大 id +1，date 从聊天记录中提取或使用当前日期。
